## 论文针对什么问题

云数据中心普遍面临**内存搁浅（memory stranding）**问题：已分配的 DRAM 因 CPU 核已全部占满而空闲，生产集群中有 25–35% 的安装内存被浪费。内存池化（memory pooling）是解决这一问题的自然方向，但当前两种主流池化技术各自存在不足：

- **CXL（Compute Express Link）** 提供基于 load/store 的低延迟透明访问，但仅限于单个 pod（8–16 插槽），且 CXL 2.0 多头设备（MHD）缺少跨主机的硬件缓存一致性，难以实现 pod 级别以上的池化。
- **RDMA** 可实现集群范围的统一内存池，但每次远程访问都伴随软件开销（缺页、内核介入或应用层重定向），延迟较高。

**分层结合 CXL 与 RDMA** 是实际的池化路径，但在 MicroVM 无服务器计算领域一直未被探索。而该场景中，**快照恢复延迟是冷启动的主要瓶颈**，迫切需要在延迟和池化范围之间取得更好的权衡。

## 提出了什么解决方案

论文提出了 **Aquifer** —— 第一个基于分层 CXL+RDMA 内存池为 MicroVM 快照提供服务的系统。其核心方案包括：

- **基于热度的快照格式**：通过分析发现快照中绝大部分页面要么为零页要么为冷页，Aquifer 消除零页，将热工作集放在快速 pod 本地 CXL 池，冷页放在集群级 RDMA 池，压缩快照体积并减少恢复关键路径上的数据搬运。
- **面向 CXL 2.0 无硬件一致性的所有权一致性协议**：允许多个主机在 CXL 2.0 多头设备上安全地共享快照，同时保证正确性。
- **基于副本的页面服务机制**：在 MicroVM 恢复前，主动将热页从 CXL 内存预安装到本地，然后异步按需从 RDMA 池拉取冷页，从而在恢复关键路径上隐藏冷页访问延迟。

## 具体是怎么做的

1. **快照镜像特征分析**  
   对典型 MicroVM 快照进行统计，揭示：
   - 绝大部分页面为零页或冷页（热工作集仅占总页面量的 5.5%），且热页地址布局高度碎片化。
   - 基于这一发现，设计**热度感知的快照格式**：存储时剔除零页，只保留非零页并标记冷热属性；热页存放在 CXL 池，冷页存放在 RDMA 池。

2. **分层内存池架构**  
   - **CXL 池**：基于 CXL 2.0 多头内存设备，挂载在同一个 pod 内的多台主机可共享，表现为 NUMA 节点，load/store 透明访问。
   - **RDMA 池**：基于单边 RDMA 操作，由集群范围内的内存节点构成，容量更大但访问延迟更高。
   - 快照按热、冷分层放置，恢复时主要从低延迟的 CXL 层读热页。

3. **CXL 共享无硬件一致性的处理：所有权一致性协议**  
   由于 CXL 2.0 多头设备不提供跨主机硬件缓存一致性，Aquifer 引入了基于所有权的软件一致性协议：
   - 快照页在 CXL 池中指定一个“所有者”节点，只有所有者可写；其他节点只能读，或在获取所有权后再写。
   - 通过明确的权限传递和无效化操作，保证多主机同时访问同一快照时不会出现数据不一致。

4. **基于副本的页面服务与恢复流程**  
   - 创建快照时，Aquifer 遍历 MicroVM 内存，识别零页并剔除，区分冷热页后分别写入 CXL 和 RDMA 池。
   - 恢复时：
     - **预安装阶段**：从 CXL 池将热页**直接拷贝**到目标主机本地内存（或 MicroVM 地址空间），无需缺页陷入。
     - **启动 MicroVM**：恢复执行。
     - **异步按需拉取冷页**：当 MicroVM 访问到尚未载入的冷页时，通过缺页从 RDMA 池异步拉取，不阻塞热路径。
   - 这种方式将快照恢复关键路径上的数据移动限定为低延迟的本地 CXL 读取，大幅缩短冷启动时间。

5. **实现**  
   Aquifer 与 Firecracker MicroVM 监视器集成，使用 CloudLab 平台模拟 CXL+RDMA 硬件环境进行评估。

## 取得了什么效果

在模拟的 CXL+RDMA 硬件上，对九个无服务器工作负载的评估结果：

- **端到端调用加速**：几何平均端到端延迟比原生 Firecracker 快 **2.2 倍**，比优化过的 FaaSnap 快 **1.3 倍**（摘要中提及 1.1 倍，可能与基准版本有关，正文为 1.3 倍）。
- **并发扩展能力**：可支持同时恢复最多 **32 个 MicroVM**。
- 快照体积因剔除零页而显著缩小，分层放置热页使得恢复时关键路径数据读取量大幅减少，直接转化为冷启动延迟的下降。

## 旁观者视角的问题与不足

- **硬件依赖性评估不足**：所有实验均基于模拟的 CXL+RDMA 环境，缺乏真实 CXL 2.0 硬件、交换机延迟特性和信号完整性的实测验证，结果是否能在真实部署中复现存疑。
- **一致性协议开销未量化**：所有权一致性协议引入了显式的权限传递与无效化，可能在高并发、频繁写共享快照的场景中产生额外延迟，文中未对此类开销进行深入分析或对比（如与纯 RDMA 方式的软件开销比较）。
- **热度定义和适应性**：分析来自特定工作负载，热页判定（如 5.5% 热集）可能不具备普适性；如何动态识别热集并在运行时调整分层策略未讨论，若工作负载改变，分层效果可能下降。
- **故障和可靠性处理缺失**：CXL 设备故障或 RDMA 链路断开时如何保证快照恢复的正确性与可用性，论文未涉及。
- **与 CXL 3.0 共享内存模型的对比不足**：没有分析与未来具备硬件一致性、多级共享能力的 CXL 3.0 方案的差异，Aquifer 的软件一致性是否是必要的长期方案。

## 值得继续追踪的点

- **真实 CXL 硬件上的验证**：在真正的 CXL 2.0 设备及交换机上复现性能结果，尤其需要测量 CXL 访问的尾部延迟和信号完整性对可用性的影响。
- **热页识别的在线化与自适应策略**：研究在 MicroVM 生命周期内动态重新识别热页并调整 CXL 与 RDMA 间数据分布的方法，提升对不同应用变化的适应能力。
- **一致性协议的优化与形式化验证**：探索更高效的所有权传递策略，减少一致性流量；对该协议进行正确性形式化验证，以确保在边界情况下的安全性。
- **混合负载下的内存池化性能**：评估 Aquifer 在多种 MicroVM 混合运行、CXL 池动态争用时的表现，以及是否能够支持一般虚拟机或容器场景。
- **集成到实际无服务器平台**：将 Aquifer 与 AWS Lambda、Azure Functions 等现有基础设施的快照机制结合，评估在真实生产环境中的冷启动改善。
- **CXL 3.0 出现后的架构演进**：跟踪 CXL 3.0 的多层交换和共享一致性能力，探索 Aquifer 在硬件一致性可用时能否简化或增强其分层设计。
